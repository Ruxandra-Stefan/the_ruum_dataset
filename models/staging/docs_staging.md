{% docs stg_users %}
One row per registered user. Email duplicates are preserved for referential integrity and flagged
via `is_duplicate_email` rather than silently dropped — downstream consumers should filter on
that column if they need a deduplicated user list.
{% enddocs %}

{% docs stg_locations %}
One row per user location covering US and Canada. Extends the raw location data with derived
fields for currency, exchange rate (sourced from the `exchange_rates` seed), supply chain
eligibility, and Quebec regulatory compliance.
{% enddocs %}

{% docs stg_orders %}
One row per order placed by a user. Includes the full order lifecycle status — see
`col_order_status` for valid values and their meanings.
{% enddocs %}

{% docs stg_order_items %}
One row per line item within an order. Numeric columns (`quantity`, `unit_price`, `discount`,
`line_total`) are explicitly cast at this layer so downstream mart models can rely on consistent
types without re-casting.
{% enddocs %}

{% docs stg_products %}
One row per product in the catalogue. Price reflects the current list price in USD at the time
of the dbt run — historical prices at purchase time are captured in `stg_order_items.unit_price`.
{% enddocs %}

{% docs stg_user_events %}
One row per user interaction event (sign-up, page view, add-to-cart). Quebec users are flagged
via `is_quebec_event` by joining to `stg_locations` through `stg_users` — use this flag to
ensure compliance-sensitive event processing for that region.
{% enddocs %}

{% docs col_is_duplicate_email %}
True when this email address appears more than once in the source data — i.e., a user registered
more than once with the same email. Determined by `ROW_NUMBER() OVER (PARTITION BY email ORDER BY
created_at)`: the first registration is `false`, all subsequent ones are `true`. All records are
kept; no rows are dropped.
{% enddocs %}

{% docs col_exchange_rate_to_usd %}
Exchange rate from the location's local currency to USD. Sourced from the `exchange_rates` seed
file — update that file and re-run `dbt seed` to change the rate. A `not_null` test and the
`assert_quebec_transparent_pricing` singular test both guard this column: a null here means the
seed join failed, which would corrupt any CAD-to-USD revenue calculation downstream.
{% enddocs %}

{% docs col_is_excluded_supply_chain %}
True when the location is ineligible for standard fulfillment due to logistics complexity.
Currently excludes: Alaska (`state = 'AK'`), Hawaii (`state = 'HI'`), and Puerto Rico
(`zip_code LIKE '009%'`). Check `supply_chain_exclusion_reason` for the human-readable
explanation. Orders to excluded locations should be routed through the specialist fulfilment
workflow before dispatch.
{% enddocs %}

{% docs col_is_quebec_compliant_required %}
True for Quebec province (`country_code = 'CA' AND state = 'QC'`) only. When true, two
regulatory frameworks apply: Bill 101 (Charter of the French Language) — all customer-facing
communications must support French; and PIPEDA — GDPR-like privacy controls are required.
See `quebec_compliance_notes` for the full documentation string, and the singular tests
`assert_quebec_transparent_pricing` and `assert_quebec_language_requirements` for automated
compliance checks.
{% enddocs %}

{% docs col_marketing_language_requirement %}
The language(s) that marketing channels must use when communicating with users at this location.
Derived from country and province under Bill 101 (Charter of the French Language):

| Value | Applies to | Requirement |
|---|---|---|
| `EN` | All US locations; non-Quebec Canada | English-only marketing is compliant |
| `EN,FR` | Quebec (`state = 'QC'`) | Both English **and** French are legally required — Bill 101 mandates French availability in all commercial communications |

Use this column to gate content delivery, email templates, and ad copy. Quebec users must receive
French-language versions of all marketing materials. The singular test
`assert_quebec_language_requirements` verifies that this flag is set correctly for all QC
locations.
{% enddocs %}

Raw e-commerce data for The Ruum — a furniture and home goods retailer operating across the US
and Canada. Data is loaded from CSV files generated by `generate_data.py` and stored in the
DuckDB `main` schema. This source is the single point of entry for all staging models; no

{% docs src_users %}
Raw user registration records. One row per registration event — duplicate emails are possible
and intentional (a user may register more than once). The staging model `stg_users` adds the
`is_duplicate_email` flag to surface these without dropping rows. Relationships to `locations`
are enforced at this layer via a foreign key test.
{% enddocs %}

{% docs src_locations %}
Raw geographic data for user locations covering the US and Canada. This table is the source for
both the `stg_locations` staging model and the `snap_locations` snapshot. It contains only the
raw geographic fields — all derived compliance, currency, and supply chain columns are computed
in staging.
{% enddocs %}

{% docs src_products %}
Raw product catalogue for The Ruum's furniture range. Categories are limited to Living Room,
Bedroom, and Kitchen. Prices are list prices in USD at the time of data generation — the price
a customer actually paid is captured in `order_items.unit_price`, which may differ due to
promotions or time-of-purchase pricing.
{% enddocs %}

{% docs src_orders %}
Raw order header records. One row per order — line items are in `order_items`. Order status
progresses through a defined lifecycle; `pending_refund` is a valid terminal-adjacent state
for delivered orders where the customer has initiated a return.
{% enddocs %}

{% docs src_order_items %}
Raw line items for each order. `line_total` is pre-computed in the source as
`quantity * unit_price * (1 - discount)`. The staging model re-casts all numeric columns
to explicit types (`integer` for quantity, `double` for monetary values) — do not rely on
inferred types from this source directly.
{% enddocs %}

{% docs src_user_events %}
Raw behavioural event stream capturing user interactions during browsing and checkout. Three
event types are recorded: `user_sign_up`, `page_view`, and `add_to_cart`. `product_id` is
nullable — it is null for events on non-product pages (e.g. homepage, checkout, sign-up).
High-volume table; the incremental model `stg_user_events_incr` is preferred over the view
for production workloads.
{% enddocs %}

{% docs col_order_status %}
Current fulfillment status of the order. One of the following values:

| status | definition |
|---|---|
| `pending` | Order placed, not yet shipped |
| `shipped` | Order has been dispatched to carrier, not yet delivered |
| `delivered` | Order confirmed received by the customer |
| `cancelled` | Order cancelled before fulfillment |
| `pending_refund` | Order delivered but customer has requested a refund — awaiting resolution |

`pending_refund` orders must be excluded from confirmed revenue totals but included in gross
merchandise value (GMV) until the refund is processed. Filter on this status in any revenue
report to avoid overstating income.
{% enddocs %}

{% docs quebec_compliance_overview %}
## Quebec Regulatory Compliance

Locations where `is_quebec_compliant_required = true` are subject to two regulatory frameworks:

### Bill 101 — Charter of the French Language
All customer-facing communications, including marketing, product descriptions, invoices, and
support, must be available in French. The `marketing_language_requirement` column will be
`EN,FR` for these locations. The singular test `assert_quebec_language_requirements` validates
this is correctly flagged in the data.

### Consumer Protection Act — Transparent Pricing
Prices must be displayed in the local currency (CAD). The `currency_code` column must be `CAD`
and `exchange_rate_to_usd` must be a positive value for all Quebec locations. The singular test
`assert_quebec_transparent_pricing` validates this automatically on every dbt run.

### PIPEDA — Privacy
GDPR-equivalent privacy controls apply. Do not use Quebec user data for profiling or targeted
advertising without explicit consent. See `quebec_compliance_notes` on each location row for
the full compliance documentation string.

**Columns that drive compliance logic in this project:**

| Column | Model | Purpose |
|---|---|---|
| `is_quebec_compliant_required` | `stg_locations` | Master compliance flag for QC |
| `marketing_language_requirement` | `stg_locations` | `EN,FR` gates bilingual content delivery |
| `currency_code` | `stg_locations` | Must be `CAD` for transparent pricing |
| `exchange_rate_to_usd` | `stg_locations` | Must be positive; guards CAD→USD conversion |
| `quebec_compliance_notes` | `stg_locations` | Human-readable regulatory summary |
| `is_quebec_event` | `stg_user_events_incr` | Flags events from QC users for compliant processing |
{% enddocs %}
